# System Architecture

This document explains how the Cisco Configuration Documentation System works.

```
┌─────────────────────────────────────────────────────────────────────┐

┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│   configs/   │      │  automation/ │      │   output/    │
│              │─────▶│              │─────▶│              │
│  *.txt files │      │  Processing  │      │  *.md files  │

                             │

                    ┌────────┴────────┐
```

## Component Details

### 1. File Watcher ([automation/watcher.js](automation/watcher.js))

**Purpose**: Monitor the `configs/` directory for new or changed configuration files.

**Technology**: `chokidar` (cross-platform file watching)

**Behavior**:
- Watches `*.txt` files in `configs/`
- Debounces file changes (waits for file to stabilize)
- Prevents duplicate processing
- Spawns processor for each detected file

**Flow**:
```
File Added/Changed
    ↓
Debounce (2 seconds)
    ↓
Check if already processing
    ↓
Spawn processor.js as child process
    ↓
Monitor completion
```

### 2. Configuration Processor ([automation/processor.js](automation/processor.js))

**Purpose**: Process a single configuration file and generate documentation.

**Main Functions**:
1. **Read Config**: Load the configuration file
2. **Extract IOS Version**: Parse version from config text
3. **Build Prompt**: Load template and inject config content
4. **Call LLM**: Send to local LLM for analysis
5. **Save Documentation**: Write markdown to output/
6. **Git Commit**: Commit changes with metadata

**Processing Pipeline**:
```
Input: configs/SWITCH-01.txt
    ↓
Extract IOS Version (e.g., "15.2")
    ↓
Load Prompt Template
    ↓
Replace {{CONFIG_CONTENT}}, {{IOS_VERSION}}, {{FILENAME}}
    ↓
Send to LLM API
    ↓
Receive Markdown Documentation
    ↓
Save to output/SWITCH-01.md
    ↓
Git Add + Commit
    ↓
Output: Generated documentation
```

### 3. MCP Server ([mcp-server/server.js](mcp-server/server.js))

**Purpose**: Provide the LLM with access to Cisco IOS documentation.

**Protocol**: Model Context Protocol (MCP)

**Available Tools**:

| Tool | Purpose | Example |
|------|---------|---------|
| `search_command` | Look up command documentation | `interface`, `ip address` |
| `get_feature_docs` | Get feature explanations | `VLAN`, `OSPF`, `STP` |
| `validate_syntax` | Validate command syntax | Check if command is valid |
| `explain_config_section` | Explain config blocks | Interpret complex sections |

**Data Sources**:
- Local cache: `docs_cache/basic_commands.json`
- Extensible to fetch from Cisco.com
- Can be enhanced with additional documentation

**Communication**:
```
LLM ←─ stdio ─→ MCP Server
         ↕
    [Search, Lookup,
     Validate, Explain]
         ↕
  docs_cache/*.json
```

### 4. Local LLM

**Supported Platforms**:
- **Ollama**: Recommended, easy setup
- **LM Studio**: GUI-based, user-friendly
- **Text Generation WebUI**: Advanced users
- **Any OpenAI-compatible API**

**Requirements**:
- Minimum 8GB VRAM (for 7B models)
- 16GB VRAM recommended (for better models)
- Context window: 8K+ tokens
- Tool/function calling support (optional but recommended)

**API Format**:
```json
POST /v1/chat/completions
{
  "model": "llama3.1:8b",
  "messages": [
    {"role": "system", "content": "You are a Cisco expert..."},
    {"role": "user", "content": "Analyze this config..."}
  ],
  "temperature": 0.7,
  "max_tokens": 8000
}
```

### 5. Git Integration

**Purpose**: Version control for generated documentation.

**Features**:
- Automatic initialization
- Commits per documentation update
- Detailed commit messages with metadata
- Optional auto-push to remote

**Commit Format**:
```
Update documentation: SWITCH-01.txt

Generated from: SWITCH-01.txt
Timestamp: 2024-11-25T14:30:00.000Z
Auto-generated by Cisco Config Documentation System
```

## Data Flow

### Complete Processing Flow

```
1. User adds file: configs/CORE-SW-01.txt
                           ↓
2. Watcher detects change
                           ↓
3. Watcher spawns: node processor.js configs/CORE-SW-01.txt
                           ↓
4. Processor reads config file
                           ↓
5. Processor extracts IOS version: "15.2"
                           ↓
6. Processor loads prompt template
                           ↓
7. Processor builds final prompt with config content
                           ↓
8. Processor sends HTTP POST to LLM API
                           ↓
9. LLM receives prompt and config
                           ↓
10. LLM uses MCP tools to look up commands:
    - search_command("interface")
    - get_feature_docs("VLAN")
    - search_command("spanning-tree")
                           ↓
11. MCP Server returns documentation from cache
                           ↓
12. LLM generates markdown documentation
                           ↓
13. Processor receives LLM response
                           ↓
14. Processor saves: output/CORE-SW-01.md
                           ↓
15. Processor commits to Git:
    - git add output/CORE-SW-01.md
    - git commit -m "Update documentation: CORE-SW-01.txt"
                           ↓
16. Process complete! ✅
```

## Error Handling

### File Watcher Errors
- File access denied → Skip and log error
- File too large → Process anyway (LLM will truncate)
- Duplicate processing → Blocked by lock set

### Processor Errors
- Config file not found → Exit with error
- IOS version not found → Continue with "Unknown"
- LLM connection failed → Exit with detailed error message
- Git commit failed → Log warning but continue

### MCP Server Errors
- Command not found → Return "not in cache" message
- Documentation missing → Suggest adding to cache
- Server crash → LLM continues without MCP tools

## Scalability Considerations

### Single Configuration
- Processing time: 30-120 seconds (depending on model and config size)
- Memory usage: Depends on LLM model size

### Batch Processing
- File watcher processes sequentially (prevents resource overload)
- Lock mechanism prevents duplicate processing
- Each file is independent (parallel processing possible with modifications)

### Large Configurations
- Configurations up to ~4000 lines are handled well
- Larger configs may require:
  - Increased `max_tokens` in config.json
  - Larger context window in LLM
  - Potential splitting into sections

## Extension Points

### 1. Add More Documentation

Edit `docs_cache/basic_commands.json`:
```json
{
  "new_command": {
    "description": "...",
    "syntax": "...",
    "examples": [...]
  }
}
```

### 2. Add New MCP Tools

Edit `mcp-server/server.js` → `setupToolHandlers()`:
```javascript
{
  name: "new_tool",
  description: "...",
  inputSchema: {...}
}
```

### 3. Customize Output Format

Edit `automation/prompts/analysis_template.txt` to change:
- Section order
- Level of detail
- Markdown formatting
- Analysis criteria

### 4. Add Pre/Post Processing

Modify `processor.js`:
- Add preprocessing before LLM call
- Add postprocessing after LLM response
- Generate additional outputs (PDF, HTML, etc.)

### 5. Integrate with External Systems

Add to `processor.js`:
- Send notifications (email, Slack, Teams)
- Upload to documentation portal
- Trigger CI/CD pipelines
- Update CMDB or network inventory

## Security Considerations

### Sensitive Data
- Configurations may contain:
  - Passwords (even encrypted)
  - IP addresses
  - Network topology information
- Keep repository private
- Review output before sharing

### MCP Server
- Runs locally (no external calls)
- Only reads from local cache
- Can be extended to fetch from Cisco (add rate limiting)

### Git Commits
- May commit sensitive config data
- Use `.gitignore` for sensitive files
- Consider using private repository

### LLM
- All processing is local (no data sent to cloud)
- Ensure LLM is properly secured
- No API keys needed for local LLMs

## Performance Optimization

### Speed Up Processing
1. Use smaller LLM model (7B instead of 70B)
2. Reduce `max_tokens` in config.json
3. Simplify prompt template
4. Use GPU acceleration for LLM
5. Disable Git integration if not needed

### Reduce Resource Usage
1. Use quantized models (Q4, Q5)
2. Limit concurrent processing (already done)
3. Clean up `docs_cache` periodically
4. Use streaming responses (requires code modification)
